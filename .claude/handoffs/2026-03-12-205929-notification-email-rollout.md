## Codebase Understanding

### Architecture Overview

The email rollout is now centered on a unified envelope/logging approach instead of ad hoc per-event sends. Backend routes build a preview context, resolve recipients, and then call `email_service.send_rendered_email(...)` or `email_service.log_attempt(...)`. Author recipient resolution is centralized through the resolver/orchestrator path so corresponding authors become `To`, other authors become `CC`, and `journals.public_editorial_email` becomes default `CC` + `Reply-To`. Reviewer and workflow emails use the same envelope concepts even when their templates are still inline-rendered. Frontend editor flows currently consume these APIs through `EditorApi` wrappers and compose dialogs/modals; reviewer compose UI is partially upgraded, invoice UI now has a basic send action, but author manual-email UI is still mostly unimplemented.

### Key Patterns Discovered

- Every new manual email flow should follow the same backend pattern:
  1. build context
  2. resolve envelope
  3. return preview
  4. send via `send_rendered_email`
  5. external fallback via `log_attempt`
- Do not block manuscript workflow on email delivery. Email status and workflow status are intentionally decoupled.
- Do not revert unrelated dirty worktree files. There is concurrent production-SOP work in this branch.
- For frontend editor compose flows, the current codebase favors adding narrow `EditorApi` methods plus page-local state instead of introducing a generic global mail composer abstraction.
- Reviewer compose dialog uses comma-separated string inputs in UI and converts them to arrays before API calls. Reuse that pattern if adding author-side compose UI.

### Decisions Made

| Decision | Options Considered | Rationale |
|----------|-------------------|-----------|
| Use unified envelope semantics instead of per-route one-off recipient logic | Keep patching each notification route vs centralize envelope behavior | Centralization reduces drift and keeps author/reviewer/journal email policy consistent |
| Keep workflow unblocked by email | Hard-gate workflow on successful send vs allow send/external/skip | User explicitly said email must never block business flow |
| Make technical revision and formal revision request non-automatic | Auto-send on state change vs manual/preview-send | User said editors must explain issues/revision details |
| Treat invoice email as manual with PDF attachment | Keep old auto invoice link email vs manual PDF send | Matches user requirement and existing invoice confirmation flow |
| First proofreading email automatic, later proofreading mail manual | Always automatic vs all manual vs split behavior | Matches business description: first proof is routine, later corrections need explanation |
| Expose reviewer `CC` and `Reply-To` in compose dialog before building author-side compose UIs | Build author-side UI first vs reviewer UI first | Reviewer compose already existed, so it was the smallest safe UI increment |
| Add minimal invoice modal send action before full invoice preview/resend UI | Build full invoice compose UI first vs simple save/send action | Gives editors an immediate usable path with low frontend surface area |
